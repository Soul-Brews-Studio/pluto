# Lesson Learned: ถามสั้นๆ → ทำ → Iterate

**Date**: 2026-01-16
**Source**: User feedback ใน session

---

## The Pattern

เมื่อทำงานกับคนที่รู้ว่าต้องการอะไร:

```
❌ Plan ยาว → ถามหลายคำถาม → รอ confirm → ทำ
✅ ถามสั้นๆ 1 คำถาม → ทำเลย → รับ feedback → iterate
```

---

## Why It Matters

1. **เร็วกว่า** - ไม่เสียเวลา plan สิ่งที่อาจเปลี่ยน
2. **Feedback จริง** - เห็นของจริงแล้วค่อย feedback ดีกว่าคุยในอากาศ
3. **Trust** - แสดงว่าเชื่อใจว่า user รู้ว่าต้องการอะไร

---

## Applied Today

```
❌ ผมทำ: สร้าง 2D demo ก่อน → ถูก reject → ทำใหม่ 3D
✅ ควรทำ: "2D หรือ 3D ครับ?" → ทำเลย

❌ ผมทำ: Copy files → ถูกบอก "ย้ายสิ"
✅ ควรทำ: "ย้ายหรือ copy?" → ทำเลย
```

---

## Key Insight

> "ถามสั้นๆ ก็พอ ไม่ต้อง plan ยาว"

คำสั้นๆ นี้บอกหลายอย่าง:
- User เป็น action-oriented
- Over-planning = เสียเวลา
- Trust the iteration process

---

## Checklist สำหรับครั้งหน้า

- [ ] ถาม 1 คำถามก่อนทำ (ถ้าไม่แน่ใจ)
- [ ] ทำเลย ไม่ต้องรอ confirm ยาว
- [ ] รับ feedback → pivot เร็ว
- [ ] ไม่สมมติ → ถามสั้นๆ แทน
